对，这就对了。
**Sylphos 从一开始就不该写成“某个模型的项目”，而要写成“一个可插拔运行时”。**

你要的其实不是：

* openWakeWord 工程
* 某个 STT 工程
* 某个 LLM 工程

而是一个统一的：

> **Sylphos Runtime 总线架构**
> 各种模型和能力都只是“挂在总线上的插件”

这样以后你换：

* 唤醒词
* STT
* TTS
* LLM
* OpenClaw / 其他执行器
* UI 前端
* 事件来源

都不会伤筋动骨。

# 一、先把目标定死

Sylphos Runtime 的核心原则应该是这四条：

### 1. 运行时中心化

所有模块都不直接互相硬连，而是统一接入 Runtime。

### 2. 接口标准化

每一类能力都定义统一接口，不让业务逻辑依赖具体模型。

### 3. 事件驱动

模块之间靠事件流和消息传递协作，而不是互相直接调用一大串内部细节。

### 4. 插件化装配

具体用 openWakeWord、Whisper、FunASR、CosyVoice、OpenAI-compatible LLM，都是配置问题，不是架构问题。

---

# 二、你要的总线架构，最核心的一句话

Sylphos Runtime 不直接等于某个模型组合，而是：

> **一个负责接收事件、分发事件、调度插件、维护状态、执行任务的本地智能运行时。**

# 四、模块怎么分

这部分最重要。你后面写代码、写论文、画图，都靠这个。

---

## 1）Runtime Core

这是大脑，不是模型本身。

建议放在：

```text
sylphos/runtime/
```

里面分这些：

### `app.py`

运行时总入口，负责启动、关闭、装配模块。

### `lifecycle.py`

生命周期管理，初始化、健康检查、资源释放。

### `events.py`

事件总线定义，事件类型、发布、订阅。

### `context.py`

上下文管理。包括当前会话、当前状态、激活任务、音频状态等。

### `routing.py`

事件路由，把不同事件发给合适模块。

### `orchestrator.py`

任务编排。负责从“用户说了一句话”到“执行动作并反馈结果”的整条链。

### `registry.py`

插件注册中心，统一管理当前加载了哪些组件。

---

## 2）Perception Layer 感知层

建议放在：

```text
sylphos/voice/
```

这个层不是只放语音，而是所有“把外界输入变成系统事件”的能力。

可以先有：

### `wakeword/`

唤醒词引擎适配器

### `stt/`

语音识别适配器

### `audio/`

音频输入输出、麦克风管理、缓冲、流处理

以后还能扩展：

### `hotkey/`

键盘唤醒

### `remote/`

手机、网页、MQTT、HA 远程触发

### `vision/`

桌面截图理解、OCR、屏幕状态分析

---

## 3）Cognition Layer 认知层

建议放在：

```text
sylphos/llm/
```

这个层负责理解、决策、生成。

包括：

### `interfaces.py`

定义 LLM 统一接口

### `schemas.py`

消息结构、工具调用结果、计划结果

### `client/`

各种模型适配器
例如：

* OpenAI-compatible
* 本地 vLLM
* Ollama
* LM Studio
* 自定义 HTTP 推理服务

### `planner.py`

把自然语言变成结构化计划

### `resolver.py`

做语义纠错、指令归一化、槽位补全

---

## 4）Execution Layer 执行层

建议放在：

```text
sylphos/executor/
```

这个层只做一件事：

> 把认知层生成的任务，真正变成机器动作

可以包括：

### `tool_executor.py`

统一工具执行入口

### `openclaw_adapter.py`

OpenClaw 适配器

### `desktop/`

桌面控制

### `system/`

系统命令、程序启动、文件操作

### `browser/`

浏览器交互、侧边栏通信

### `safety.py`

风险操作确认、白名单、黑名单、权限约束

---

## 5）Feedback Layer 反馈层

建议放在：

```text
sylphos/frontend/
sylphos/voice/tts/
```

包括：

### 语音反馈

TTS 输出

### 文字反馈

浏览器侧边栏、Web UI、日志面板

### 状态反馈

“正在听”“已识别”“正在执行”“执行失败”等状态广播

# 五、最关键的不是目录，而是接口

你说的“任何模型组件都能随便换”，本质上不是靠目录实现的，
而是靠**统一接口**实现的。

下面这套接口思想是核心。

---

## 1）唤醒词接口

```python
from __future__ import annotations
from typing import Protocol, Callable
import numpy as np


class WakeWordEngine(Protocol):
    def set_callback(self, callback: Callable[[str, float], None]) -> None: ...
    def consume(self, audio: np.ndarray) -> None: ...
    def pause(self) -> None: ...
    def resume(self) -> None: ...
    def reset(self) -> None: ...
    def close(self) -> None: ...
```

这样：

* openWakeWord 可以实现它
* Porcupine 可以实现它
* 你以后自研模型也可以实现它

---

## 2）STT 接口

```python
class STTEngine(Protocol):
    def transcribe_file(self, wav_path: str) -> str: ...
    def transcribe_pcm(self, audio, sample_rate: int) -> str: ...
    def close(self) -> None: ...
```

这样你就能换：

* Whisper
* faster-whisper
* SenseVoice
* FunASR
* sherpa-onnx
* 云端接口

---

## 3）LLM 接口

```python
class LLMEngine(Protocol):
    def chat(self, messages: list[dict], tools: list[dict] | None = None) -> dict: ...
    def close(self) -> None: ...
```

以后换：

* OpenAI-compatible
* 本地模型服务
* 远程 API
* 多模型路由

都不影响上层。

---

## 4）TTS 接口

```python
class TTSEngine(Protocol):
    def synthesize_to_file(self, text: str, output_path: str) -> str: ...
    def speak(self, text: str) -> None: ...
    def close(self) -> None: ...
```

以后换：

* Fish Audio
* CosyVoice
* Edge TTS
* GPT-SoVITS
* pyttsx3

都可以。

---

## 5）工具执行接口

```python
class ToolExecutor(Protocol):
    def execute(self, tool_name: str, arguments: dict) -> dict: ...
```

以后换 OpenClaw，或者你再加别的控制层，也不用改业务流。

---

# 六、真正的“总线”该怎么理解

你这里说的总线，不一定非得上很重的消息队列。
你当前阶段，用**进程内事件总线**就足够了。

---

## 事件模型建议

比如定义这些事件：

```python
class Event: ...
class WakeWordDetected(Event): ...
class RecordingStarted(Event): ...
class RecordingCompleted(Event): ...
class STTCompleted(Event): ...
class IntentResolved(Event): ...
class ToolExecutionRequested(Event): ...
class ToolExecutionCompleted(Event): ...
class TTSRequested(Event): ...
class UIMessageRequested(Event): ...
class ErrorOccurred(Event): ...
```

然后：

* 唤醒词模块发 `WakeWordDetected`
* 录音器收到后开始录音
* 录音完发 `RecordingCompleted`
* STT 收到后转文字，再发 `STTCompleted`
* LLM 收到后理解任务，发 `IntentResolved`
* 执行器收到后执行，再发 `ToolExecutionCompleted`
* TTS/UI 收到后反馈

这就叫总线。

---

# 七、这比“模块直接互相调用”强在哪里

因为一旦你写成：

```text
wakeword -> recorder -> stt -> llm -> executor -> tts
```

看起来简单，但以后一改就很难受：

* 想插入日志统计
* 想加安全确认
* 想加 UI 状态同步
* 想加“先展示再执行”
* 想加多轮确认
* 想加手机端接管
* 想并行触发多个反馈

都会开始牵一发动全身。

但如果是事件总线：

* 多加一个订阅者就行
* 原链路不用大改

# 八、你当前这套代码，怎么平滑升级到总线架构

你现在已经有这些部件了：

* `AudioHub`
* `WakeWordConsumer`
* `CommandRecorder`
* `VoiceController` 
* 入口 `run_audiohub_wakeword.py` 

这其实已经是第一步了。

## 当前结构的问题

你现在的 `VoiceController` 还是“流程硬编码控制器”：

* 收到唤醒
* pause wakeword
* start_recording
* 录音完成
* reset wakeword

这个阶段没问题，但它还是偏“直连”。

---

## 下一步升级方法

把 `VoiceController` 改成：

### 不直接控制所有模块

而是：

* 接收事件
* 发布事件
* 由编排器处理流程

也就是：

### 现在

```text
WakeWordConsumer -> VoiceController -> Recorder
```

### 以后

```text
WakeWordConsumer -> EventBus.publish(WakeWordDetected)
RuntimeOrchestrator -> 订阅事件 -> 控制 Recorder
```

这就是从 demo 到 runtime 的分水岭。

# 十、配置层也必须可插拔

你后面千万别把模型写死在代码里。
建议所有引擎都配置化。

例如：

```python
WAKEWORD_PROVIDER = "openwakeword"
STT_PROVIDER = "faster_whisper"
LLM_PROVIDER = "openai_compatible"
TTS_PROVIDER = "cosyvoice"
TOOL_PROVIDER = "openclaw"
```

然后各自有工厂：

```python
wake_engine = create_wakeword_engine(config)
stt_engine = create_stt_engine(config)
llm_engine = create_llm_engine(config)
tts_engine = create_tts_engine(config)
tool_executor = create_tool_executor(config)
```

这样以后你切换模型，只改配置，不改业务主流程。
